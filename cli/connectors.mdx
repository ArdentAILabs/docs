A connector links Ardent to one Postgres source database. You need a healthy connector before creating branches.

For production databases, start with preflight. It checks the source and prints the SQL to fix missing grants **without** creating a connector row or storing credentials.

### ardent connector preflight

Validate a Postgres source before creating a connector.

```bash
ardent connector preflight postgresql 'postgresql://user:***@host:5432/mydb'
```

Preflight runs the same source-readiness gate as connector creation, but it does not persist anything. It prints:

- connection and authentication checks
- writer/read-replica check
- logical replication settings
- replication slot and WAL sender capacity
- `wal2json` availability
- table-read and schema-write checks
- duplicate-source detection
- a generated grant script when Ardent can infer the provider and username

Use `--schemas` to generate schema-specific grant SQL:

```bash
ardent connector preflight postgresql 'postgresql://user:***@host:5432/mydb' \
  --schemas public,billing
```

For BYOC/private-network sources, route the check through the same environment and private connection you will use for connector creation:

```bash
ardent connector preflight postgresql '<url>' \
  --byoc \
  --environment-id env_123 \
  --private-link-id epl_123
```

If an environment has an active private connection, Ardent requires `--private-link-id` so preflight and connector creation do not accidentally test through the public path.

### ardent connector create

Create a connector after preflight passes.

```bash
ardent connector create postgresql 'postgresql://user:***@host:5432/mydb'
```

**Shell quoting:** Wrap the connection URL in **single quotes**. Otherwise the shell may treat characters in real strings (for example `?`, `&`, `#`, `!`, `$`, parentheses, or spaces inside passwords) as globs, history expansion, or variable substitution, and the command can fail or connect to the wrong place.

Give the connector a name with `-n`:

```bash
ardent connector create postgresql 'postgresql://user:***@host:5432/mydb' -n prod-db
```

<Accordion defaultOpen={false} title="Example output">
```text
✓ Connector created and ready
  ID: conn_abc123
  Ready to branch:
  ardent branch create <name>
```
</Accordion>

Initial setup can take minutes to hours depending on source size, write rate, and network throughput. Once the connector is healthy and caught up, branch creation is fast and independent of source database size.

**BYOC / private-network connector:**

```bash
ardent connector create postgresql '<url>' \
  --byoc \
  --environment-id env_123 \
  --private-link-id epl_123
```

**Replica identity decisions:**

Some tables cannot safely replicate UPDATE/DELETE events unless Postgres can identify the affected row. If Ardent finds tables without a primary key, valid unique NOT NULL index, or `REPLICA IDENTITY FULL`, the CLI asks what to do.

For non-interactive flows, pass a complete decision file:

```bash
ardent connector create postgresql '<url>' \
  --replica-identity-decisions ./replica-identity-decisions.json
```

Decision values are:

- `exclude` — do not replicate this table yet
- `add_pk` — you will add a primary key or valid unique key on source before retrying
- `replica_identity_full` — opt into `REPLICA IDENTITY FULL` for this table

You can also accept safe defaults non-interactively. Undecided tables are excluded:

```bash
ardent connector create postgresql '<url>' --accept-replica-identity-defaults
```

<Warning>
  `REPLICA IDENTITY FULL` increases WAL volume for UPDATE and DELETE on that table. Use it deliberately; Ardent never changes replica identity on your source automatically.
</Warning>

### ardent connector list

List all connectors in the current project. In the terminal, the **active** connector is **highlighted in green** and prefixed with `*`; other rows use default/dim text. `●` means healthy, `○` means not healthy. The service type prints on the line below each name. Warnings appear in yellow with a count on the name line and full text indented below.

```bash
ardent connector list
```

### ardent connector switch

Switch the active connector without changing projects.

```bash
ardent connector switch staging-db
```

<Accordion defaultOpen={false} title="Example output">
```text
Switched from 'prod-db' to 'staging-db'
```
</Accordion>

### ardent connector retry-setup

Retry branching-engine setup for a connector that validated but did not finish setup.

```bash
ardent connector retry-setup prod-db
```

If retry needs replica identity decisions, pass the same flags as `connector create`:

```bash
ardent connector retry-setup prod-db \
  --replica-identity-decisions ./replica-identity-decisions.json
```

### ardent connector update

Update connector configuration.

Drop selected source extensions from branches:

```bash
ardent connector update prod-db --drop-extensions pgmq,supabase_vault
```

Pass an empty string to clear the list:

```bash
ardent connector update prod-db --drop-extensions ''
```

### ardent connector delete

Delete a connector.

```bash
ardent connector delete staging-db
```

By default, Ardent waits for in-flight WAL/Kafka work to drain so replicated changes are not abandoned. Use `--force` only when you intentionally want to skip that wait.

```bash
ardent connector delete staging-db --force
```
