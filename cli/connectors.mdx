For production databases, start with preflight. It checks the source and prints the SQL to fix missing grants **without** creating a connector or storing credentials.

### ardent connector preflight

Validate a Postgres source before creating a connector.

```bash
ardent connector preflight postgresql 'postgresql://user:[YOUR-PASSWORD]@host:5432/mydb'
```

Preflight runs a series of checks so you know whether Ardent can create a working connector for your database. If anything's missing, you'll see exactly what to fix. It does not store credentials or create anything. The checks cover:

- connection and authentication
- whether you're connected to a writer (not a read replica)
- logical replication settings
- replication slot and WAL sender capacity
- `wal2json` plugin availability
- Ardent's permission to read your tables and create its metadata schema
- whether this server is already connected to Ardent

Preflight detects your Postgres provider and the permissions your user has. If any required permissions are missing, it generates a grant script you can paste back to fix them in one step.

If your tables live in schemas other than `public`, pass `--schemas` so preflight checks permissions on those schemas too:

```bash
ardent connector preflight postgresql 'postgresql://user:[YOUR-PASSWORD]@host:5432/mydb' \
  --schemas public,billing
```

For BYOC/private-network sources, route the check through the same environment and private connection you will use for connector creation:

```bash
ardent connector preflight postgresql '<url>' \
  --byoc \
  --environment-id env_123 \
  --private-link-id epl_123
```

If an environment has an active private connection, Ardent requires `--private-link-id` so preflight uses the same private route the connector will.

### ardent connector create

Create a connector after preflight passes.

```bash
ardent connector create postgresql 'postgresql://user:[YOUR-PASSWORD]@host:5432/mydb'
```

**Shell quoting:** Wrap the connection URL in **single quotes**. Otherwise the shell may treat characters in real strings (for example `?`, `&`, `#`, `!`, `$`, parentheses, or spaces inside passwords) as globs, history expansion, or variable substitution, and the command can fail or connect to the wrong place.

Give the connector a name with `--name`:

```bash
ardent connector create postgresql 'postgresql://user:[YOUR-PASSWORD]@host:5432/mydb' --name prod-db
```

```text
✓ Connector created and ready
  ID: conn_abc123
  Ready to branch:
  ardent branch create <name>
```
</Accordion>

Initial setup can take minutes to hours depending on source size, write rate, and network throughput. After setup, branch creation is fast and independent of source database size.

**BYOC / private-network connector:**

```bash
ardent connector create postgresql '<url>' \
  --byoc \
  --environment-id env_123 \
  --private-link-id epl_123
```

**Replica identity decisions:**

Some tables cannot safely replicate UPDATE/DELETE events unless Postgres can identify the affected row. If Ardent finds tables without a primary key, valid unique NOT NULL index, or `REPLICA IDENTITY FULL`, the CLI asks what to do.

For non-interactive flows, pass a complete decision file:

```bash
ardent connector create postgresql '<url>' \
  --replica-identity-decisions ./replica-identity-decisions.json
```

Decision values are:

- `exclude`: do not replicate this table yet
- `add_pk`: you will add a primary key or valid unique key on source before retrying
- `replica_identity_full`: opt into `REPLICA IDENTITY FULL` for this table

You can also accept safe defaults non-interactively. Undecided tables are excluded:

```bash
ardent connector create postgresql '<url>' --accept-replica-identity-defaults
```

<Warning>
  `REPLICA IDENTITY FULL` increases WAL volume for UPDATE and DELETE on that table. Use it deliberately; Ardent never changes replica identity on your source automatically.
</Warning>

### ardent connector list

List all connectors in the current project. In the terminal, the **active** connector is **highlighted in green** and prefixed with `*`; other rows use default/dim text. `●` means the connector is ready to branch from; `○` means it is not. The service type prints on the line below each name. Warnings appear in yellow with a count on the name line and full text indented below.

```bash
ardent connector list
```

### ardent connector switch

Switch the active connector without changing projects.

```bash
ardent connector switch staging-db
```

<Accordion defaultOpen={false} title="Example output">
```text
Switched from 'prod-db' to 'staging-db'
```
</Accordion>

### ardent connector retry-setup

Retry setup for a connector whose preflight passed but did not finish creating.

```bash
ardent connector retry-setup prod-db
```

If the original setup needed replica identity decisions, pass the same JSON file when retrying:

```bash
ardent connector retry-setup prod-db \
  --replica-identity-decisions ./replica-identity-decisions.json
```

### ardent connector update

Update connector configuration.

Drop selected source extensions from branches:

```bash
ardent connector update prod-db --drop-extensions pgmq,supabase_vault
```

Pass an empty string to clear the list:

```bash
ardent connector update prod-db --drop-extensions ''
```

### ardent connector delete

Delete a connector.

```bash
ardent connector delete staging-db
```

By default, Ardent waits for in-progress replication work to finish so changes aren't abandoned. Use `--force` only when you intentionally want to skip that wait.

```bash
ardent connector delete staging-db --force
```
