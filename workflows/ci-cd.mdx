description: "A fresh database for every PR. Isolated, production-like, torn down after merge."
---

Every PR gets its own fresh database cloned from production, torn down after merge. No shared staging that drifts out of sync. No test runs that corrupt each other.

## GitHub Actions example

The CLI needs to authenticate before running any commands. In CI, use `ardent login --token` with a token stored in your repo secrets. This creates a local session the same way `ardent login` does interactively.

After login, create the branch. The CLI doesn't expose the URL as a structured output yet, so grab it from `ardent branch info` and append it to `GITHUB_ENV` so later steps (npm, pytest, your migration runner) see `DATABASE_URL`.

```yaml
- name: Install Ardent CLI
  run: npm install -g ardent-cli

- name: Log in to Ardent
  run: ardent login --token ${{ secrets.ARDENT_TOKEN }}

- name: Create database branch
  run: |
    NAME="pr-${{ github.event.pull_request.number }}"
    ardent branch create "$NAME"
    echo "DATABASE_URL=$(ardent branch info | awk '/URL:/{print $2}')" >> "$GITHUB_ENV"

- name: Run migrations + tests
  run: npm test

- name: Delete branch
  if: always()
  run: ardent branch delete pr-${{ github.event.pull_request.number }}
```

Use the same `DATABASE_URL` for `pytest`, `pnpm test`, or any tool your repo already uses in CI. If your framework reads a different variable, write that name to `GITHUB_ENV` instead.

## Why this beats a shared staging database

- **Isolation:** each PR can't interfere with other test runs
- **Always in sync:** branched from production, no drift
- **No idle cost:** branches auto-suspend between CI runs
- **Auto cleanup:** `if: always()` means no orphaned environments
